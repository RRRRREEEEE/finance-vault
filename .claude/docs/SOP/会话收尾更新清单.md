# 会话收尾更新清单

> **触发词**："更新你要更新的" / "更新一下" / "收尾"
> **用途**：每次工作会话结束前，AI 自主扫描所有过时的元数据文件并更新。不等人类逐个点名。

---

## 一、必做（每次）

### 1.1 进度文件
- `D:\data\金融学\阅读进度.json` — `last_updated` 时间戳、`processed_pages`、`notes_generated`、章节 `status`
- 如果涉及学习笔记补充：更新 `supplement_task` 下的章节状态

### 1.2 Git 提交与推送

**版本号规则（人类控制）**：
- **版本号由人类决定**，AI 不自行打版本号
- **Git 提交只在人类说"更新"或指定版本号之后执行**——AI 不在每章结束后自行 commit/push
- 人类说"更新"时，AI 列出本次会话所有未提交改动，建议版本号（2-3个选项），等人类确认后提交
- 参考建议规则：
  - 一次新会话做实质性工作 → 建议小版本（v2.4 → v2.5）
  - 人类说"更新" → 建议补丁（v2.4.6 → v2.4.7）
  - 人类说"大版本" → 建议大版本（v2.x → v3.0）
- 注意：`模板版本.md` 的版本号是笔记模板格式的版本，和 git tag 是两套独立体系

**人类确认后执行**：
```bash
cd /d/data/金融学
git add -A
git commit -m "v{版本号} · {一句话概括}"
git tag "v{版本号}"
git push
git push origin "v{版本号}"
```

- [ ] 版本号由人类确认（AI 只提供建议）
- [ ] 已打 tag，已分别 push（tag 需单独推送）

### 1.3 变更日志
- `变更日志.md` — 记录本次会话做了什么、发现问题、修改了哪些规则/模板
- 格式：`## YYYY-MM-DD · 一句话概括`，下面列要点

## 二、涉及时必做

| 做了什么 | 更新什么 |
|----------|----------|
| 新建/补充/重评级了笔记 | 对应的章节索引文件（`{课程} ch{XX} {标题}.md`） |
| 修改了某笔记的内容 | 该笔记的 `source` 字段、`importance`（如从壳升级） |
| 笔记新增了 `[[链接]]` | 被链接笔记的 `📖 相关术语` 里补回链 |
| 调整了某个 importance 星级 | 同类空壳/内容笔记星级是否统一 |
| 碰到了新问题并解决 | `问题解决与标准化.md` |
| 发现了检查清单覆盖不了的质量问题 | `笔记检查清单.md` 新增检查项 |
| 形成了新的操作模式/约定 | `AI协作指令_v2.md` 迭代日志 + `人机协作模式手册.md` |
| 人类提了新偏好/规则调整 | `用户指令与偏好记录.md` |
| 人类粘贴了图片让你重命名 | 1) 重命名+移入 `999-Attachments/` 2) 更新笔记 `![[引用]]` + 图形说明 3) 提醒人类删 Vault 根目录原图 |
| 进度摘要变化（完成/下个任务） | `CLAUDE.md` 进度表 |
| 全库笔记总数变化 | `MEMORY.md`（项目 memory 里的数字） |

## 三、不需要更新

| 文件 | 原因 |
|------|------|
| `笔记制作SOP.md` | 操作流程稳定，除非流程本身变了 |
| `.claude/rules/always.md` | 红线不变（除非人类要求改红线） |
| `.claude/rules/vault-notes.md` | 路径和命名规则不变 |
| 临时文件（如 `_chXX_study_guide.txt`） | 用完即弃，不纳入元数据 |
| 源文件目录 | 红线：绝不碰 |

## 四、最终确认

更新完成后口头报告：
- 本次会话做了什么
- 更新了哪些文件（列清单）
- 有没有遗留的 TODO / 待人类确认项
- 下个会话的断点是什么

---

## 五、边界规则

- **不确定改不改 → 改。** 宁可多更新一个文件，不要留过时信息。
- **更新前先 Read → 再 Edit。** 不盲写。
- **全库笔记数变化 → 同步更新 `rule-changes-20260606.md`（memory）和 `变更日志.md` 和 `阅读进度.json`。**
